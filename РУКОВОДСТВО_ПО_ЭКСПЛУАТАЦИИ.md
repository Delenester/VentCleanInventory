# Руководство по эксплуатации VentClean

Система автоматизации учёта вентиляционных работ, склада и взаимодействия с клиентами.

---

## 1. Запуск системы

### Требования
- .NET 8 SDK
- SQL Server LocalDB (поставляется с Visual Studio)
- PowerShell 5.1+

### Запуск
```bash
cd VentCleanInventory
dotnet run --project VentCleanInventory.Web
```

Приложение запускается на:
- HTTP: `http://localhost:5053`
- HTTPS: `https://localhost:7018`

При первом запуске автоматически создаются 2 базы данных:
- `VentCleanInventory` — доменные данные
- `VentCleanInventoryIdentity` — пользователи и роли

Сидер заполняет БД тестовыми данными (см. раздел 4).

## 2. Бизнес-процессы

### 2.1. Жизненный цикл заявки клиента

```
Клиент создаёт заявку (New)
       ↓
Менеджер одобряет (Approved) или отклоняет (Rejected)
       ↓
Клиент подтверждает договор (ClientConfirmed)
       ↓
Диспетчер выдаёт материалы (Assigned)
       ↓
Мастер выполняет работы (InProgress)
       ↓
Диспетчер завершает работу (Completed)
```

**Подробно:**

1. **Клиент** → «Оставить заявку» → заполняет объект, адрес, тип услуги, площадь, примерную стоимость, фото/чертёж
2. **Менеджер** → «Заявки» → просматривает, устанавливает стоимость, назначает мастера, даты → «Одобрить». Генерируется номер договора.
3. **Клиент** → на главной появляется кнопка «Подтвердить договор» → заявка переходит в статус `ClientConfirmed`
4. **Диспетчер** → «Заявки» → «Выдать» → выбирает материалы со склада → статус `Assigned`
5. **Мастер** → «Мои работы» → «Добавить работу» → выбирает объект, материалы, метры, решётки → создаётся WorkLog + акт списания
6. **Диспетчер** → «Заявки» → «Завершить» → статус `Completed`. Клиенту приходит уведомление.

### 2.2. Работа мастера на объекте

1. Зайти в «Мои работы» → карточка объекта
2. Нажать «Чек-лист» → отметить выполненные пункты, добавить фото
3. После завершения → фотоотчёт загружается через кнопку загрузки фото
4. Если обнаружен брак → «Брак» → «Добавить» → описание, зона, фото

### 2.3. Складской учёт

**Приход товара** (Диспетчер):
- «Приёмка» → выбрать поставщика → заполнить номенклатуру и количество
- Товар поступает на Центральный склад

**Выдача мастеру** (Диспетчер):
- «Заявки» → «Выдать» → выбрать материалы → товар перемещается с Центрального склада в карманный склад мастера

**Списание материалов** (Мастер):
- При добавлении работы → выбрать материалы из своего склада → списываются в расход
- Автоматически создаётся акт списания (черновик)

**Утверждение акта** (Диспетчер):
- «Списание» → открыть акт → «Утвердить»

**Возврат** (Диспетчер):
- «Возвраты» → загрузить возврат от мастера

**Остатки** (Диспетчер):
- «Остатки» → посмотреть наличие на любом складе + фильтр по сроку годности

## 3. Структура интерфейса

### Левое меню (сайдбар)

Для каждой роли своё меню:

**Администратор:**
- Главная
- Номенклатура
- Поставщики
- Клиенты
- Объекты
- Склады
- Пользователи
- Бэкап

**Менеджер:**

- Выполненные работы
- Клиенты
- Заявки

**Диспетчер:**
- Главная
- Приёмка
- Заявки
- Возвраты
- Списание
- Остатки
- Выполненные работы

**Мастер:**
- Главная
- Мой склад
- Мои заявки
- Мои работы
- Брак
- Профиль

**Клиент:**
- Главная (с уведомлениями)

- Выполненные работы

**Поставщик:**
- Главная
- Мои поставки

## 4. Тестовые данные

При первом запуске сидер создаёт:

- **2 клиентские организации**: ООО «Промвентиляция», АО «ТеплоСтрой»
- **2 поставщика**: ООО «ТехноВент», ИП «ФильтрСервис»
- **3 объекта**: Административное здание, Производственный цех, Торговый центр
- **20 позиций номенклатуры**: фильтры, ремни, подшипники, вентиляторы, двигатели и т.д.
- **~60 единиц товаров на складе** с различными серийными номерами
- **6 тестовых заявок** в разных статусах (New, Approved, Completed)
- **5 выполненных работ** (WorkLogs) с материалами
- **8 складских транзакций**: приход, выдача, возврат, перемещение

При повторном запуске сидер НЕ срабатывает (проверка: `if (anyUsers && await db.Nomenclatures.AnyAsync()) return`).

Чтобы перезапустить с чистыми данными:
```sql
DROP DATABASE VentCleanInventory;
DROP DATABASE VentCleanInventoryIdentity;
```
Или используйте кнопку «Бэкап» в админ-панели.

## 5. Договор (Word)

Система генерирует договор в формате `.docx`. Доступен:
- **Менеджеру** — на странице деталей заявки («Скачать»)
- **Клиенту** — на карточке заявки и на странице подробнее

Структура договора:
1. Предмет договора
2. Цена и порядок расчётов (сумма прописью)
3. Сроки выполнения
4. Права и обязанности сторон
5. Ответственность сторон (гарантия 12 месяцев, пеня 0,1%)
6. Изменение и расторжение
7. Форс-мажор
8. Разрешение споров
9. Прочие условия
10. Юридические адреса сторон (таблица)
11. Подписи сторон (таблица)

Данные для договора берутся из:
- Заявки (номер договора, стоимость, даты, объект)
- Организации клиента (название, УНП, адрес)
- Пользователя клиента (банковские реквизиты, телефон)

## 6. Экспорт отчётов (Excel)

Менеджер может экспортировать отчёты в Excel:
- **Затраты по объектам** → `ExportCostsByObjectsExcel`
- **Динамика затрат** → `ExportCostsTrendExcel`
- **Закупки (прогноз)** → `ExportForecastPurchases`

Все отчёты доступны через боковое меню Руководителя.

## 7. Управление пользователями

### Регистрация
- Любой может зарегистрироваться через страницу `/Account/Register`
- Новый пользователь получает роль «Клиент» с `IsApproved = true`
- При регистрации заполняются банковские реквизиты

### Администрирование
- Админ может создавать/редактировать/удалять пользователей
- Изменять роли (кроме Admin и Manager — это может только менеджер)
- Сбрасывать пароли

### Политика паролей
- Минимум 8 символов
- Обязательно наличие цифры
- Заглавные/строчные буквы не обязательны

## 8. Уведомления

Система отправляет внутрисайтовые уведомления:

| Событие | Кому | Текст |
|---|---|---|
| Новая заявка | Менеджеру | «Поступила новая заявка №...» |
| Одобрение заявки | Клиенту | «Заявка одобрена. Стоимость: ... Договор: ... Мастер: ...» |
| Отклонение заявки | Клиенту | «Заявка №... отклонена. Причина: ...» |
| Регистрация клиента | Менеджеру | «Зарегистрирован новый пользователь ...» |
| Работа выполнена | Клиенту | «Работа по заявке №... завершена.» |

Непрочитанные уведомления показываются:
- Красный бейдж в сайдбаре (для менеджера и клиента)
- Блок уведомлений на главной клиента

---

## 9. Бэкап базы данных

Администратор может создать резервную копию через интерфейс:
- «Бэкап» → «Создать бэкап»
- Файл сохраняется в `wwwroot/backups/`
- **Важно**: сохраняется только доменная БД (`VentCleanInventory`). Identity-БД бэкапится отдельно через SQL Server.

## 10. Архитектура системы

```
┌─────────────────────────────────────────────────────┐
│                    ASP.NET Core 8                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │
│  │  Admin   │ │ Manager  │ │Dispatcher│ │ Master │ │
│  │  Area    │ │  Area    │ │  Area    │ │  Area  │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘ │
│  ┌──────────┐ ┌──────────┐                           │
│  │  Client  │ │ Supplier │                           │
│  │  Area    │ │  Area    │                           │
│  └──────────┘ └──────────┘                           │
├─────────────────────────────────────────────────────┤
│  Services: StockService, WriteOffActService          │
├─────────────────────────────────────────────────────┤
│  DB: VentCleanInventory (SQL Server LocalDB)         │
│  DB: VentCleanInventoryIdentity (ASP.NET Identity)   │
└─────────────────────────────────────────────────────┘
```

- **Две отдельные БД**: Identity-данные изолированы от доменных
- **EnsureCreatedAsync**: схема создаётся автоматически (без миграций EF)
- **Ограничение скорости**: 5 login/min, 30 API req/min
- **HTTPS**: принудительное перенаправление
